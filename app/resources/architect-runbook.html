The Architect's Runbook

Two playbooks side by side. Greenfield — you're starting a new engagement and need to set it up so it doesn't break in six months. Brownfield — you're taking over a codebase and team you didn't build, and need to ship without breaking it on Tuesday. The course tells you what's true; the checklist asks what's missing; the runbook is what you actually do.

For Architects · Tech Leads · Engineering Managers Length First 90 days, end-to-end Pairs with Course · Checklist · FAQ
Path A · Greenfield

Starting a new engagement

You've been named architect on a project that doesn't have a single line of code yet. Six phases, roughly 60 days from kickoff to first ship. Each phase has the question it must answer before the next phase can start.

Phase 1
Discovery — the first two weeks
Week 1–2 · Output Discovery doc

Discovery has one job: surface the questions you don't yet know to ask. The deliverable isn't a plan — it's a list of decisions that need to be made, with named owners and dates.

Run the discovery checklist end-to-end
Every O, D, E letter of CODE-CODER asked in stakeholder workshops. Use the customer/OEM columns; both perspectives matter. Don't skip the ones you "already know" — that's where the gaps hide.
  • Output · all 216 checklist items answered or marked TBD with a named owner
  • Trap · skipping checklist items because they "feel obvious." Document the obvious; the next architect won't share your context.
Build the stakeholder map
Every named person who can say no or who must be informed. Their role, their influence (high/low), their stance (champion/neutral/sceptic), and the cadence at which they need to be updated.
  • Output · RACI matrix plus a one-page political map
  • Trap · missing the "shadow stakeholder" — the person nobody invited but everyone defers to. Ask: "who would be upset if they found out about this from someone else?"
Document the constraints, not just the requirements
Requirements describe what the system should do; constraints describe what it can't do. Compliance regimes, existing vendor contracts, internal political vetoes, hard budget caps, regulator audit windows. Constraints are usually more decisive than requirements.
  • Output · constraint register, with the source for each constraint and its severity
  • Trap · treating "we've always used vendor X" as a requirement rather than a constraint. They're different in negotiation.
Phase 2
Tech stack call
Week 2–3 · Output Architecture decision record

The first irreversible decision of the engagement. Documented as an ADR (Architecture Decision Record) — the format itself forces clarity about what was decided, what was rejected, and why.

CMS / commerce / DXP — pick the central platform
If the client mandates one, document it as a constraint, not a decision. If you're choosing, weigh: team expertise, content velocity needs, integration estate, total cost over five years, exit cost. The exit cost number is usually the one nobody calculates and the one that hurts most later.
  • Output · ADR-001 platform decision, with at least three options compared
  • Trap · letting "the team knows X" outweigh "X is wrong for the workload." Skills can be hired.
Front-end stack — and the headless question
If the central platform is AEM: AEM HTL, EDS, or fully headless via CF/GraphQL? If it's commerce: storefront framework (Next.js, Hydrogen, custom)? The decision sets the testing strategy, the perf budget, and the team shape. Decide once, document hard.
  • Output · ADR-002 front-end stack, with perf budget set against it
Hosting + the cloud provider
Often constrained by the customer's existing enterprise agreements. Don't fight it without strong reason. Match the choice to where the data must reside (sovereignty), where the team's expertise is, and what managed services you need.
  • Output · ADR-003 hosting + provider
  • Trap · "We've always used AWS" — verify the constraints still hold. Customer enterprise agreements change.
Observability stack — pick before code, not after
Logs, metrics, traces. APM vendor (or open-source). Tag manager for client-side. SLO definitions per service. Picking observability before the first line of code is what lets you instrument from day one rather than retrofit.
  • Output · ADR-004 observability stack + SLI/SLO drafts
AEM EDS vs AEMaaCS — the authoring model first
Settle the authoring paradigm before any technical architecture is drawn. Document-based authoring (EDS with Drive or SharePoint) and traditional AEM component authoring are not interchangeable delivery modes — they carry fundamentally different performance contracts, author personas, and personalisation ceilings. The decision criteria are: content velocity needed, whether the primary author is a marketer or a developer, personalisation depth required, SEO and Core Web Vitals targets (EDS has structural edge-rendering advantages), and long-term flexibility for the client's team to own the platform. Make the call here; do not let it drift into the frontend architecture step.
  • Output · ADR recording the chosen authoring model and explicitly ruling out the three alternatives considered, with one-line rationale for each rejection
  • Trap · choosing EDS because it is newer without modelling the personalisation gap. EDS is statically rendered at the edge — server-side personalisation requires a separate pattern (Target edge delivery, client-side hydration, or a KV store). If personalisation depth is high, EDS introduces complexity that AEMaaCS solves natively; picking EDS first and bolting on personalisation second is the single fastest way to blow the delivery budget.
Commerce topology — headless or composable, B2B or B2C
If commerce is in scope, agree the topology before the platform is named. Headless means a single commerce engine behind an API; composable means headless plus independently deployed microservices for cart, checkout, and catalogue. Two questions force the decision: "Does the product catalogue exceed 100,000 SKUs?" — yes means the solution needs a robust search and browse API, not CMS search — and "Are there B2B buyers with account-specific pricing?" — yes means an account hierarchy and contract pricing model must live in the commerce platform, not patched in from the CRM. B2C engagements add a third constraint: PSP-hosted fields must be scoped at this stage to contain PCI surface area, not retrofitted after the checkout is built.
  • Output · Commerce topology ADR covering: headless vs composable choice, B2B/B2C distinction with specific constraints called out, PSP selection rationale and PCI scope boundary, catalogue scale and the search/browse API pattern required
  • Trap · selecting the commerce platform before the topology is agreed. Magento is a poor fit for a composable architecture; Commercetools is a poor fit for teams expecting a UI-driven, monolithic operator experience. Topology first, platform second — reversing that order costs a full re-platforming conversation at month three.
Phase 3
Initial architecture
Week 3–4 · Output Architecture diagrams + integration map

The deployment 10-layer diagram, instantiated for this engagement. Every box has a vendor or service name, every line has a protocol and an auth model. If anything is "TBD" at the end of this phase, it's a risk on the register.

Draw the deployment topology
DNS · CDN · WAF · LB · web tier · app tier · data tier · queues · API gateway. Plus the dispatcher/proxy tier if AEM. Mark every TLS boundary, every authentication boundary, every cache. Annotate with the vendor/service name.
  • Output · production topology diagram, plus non-prod simplification
Map every integration
Each third-party touchpoint as its own row: system name, auth model, data direction (in/out/both), volume estimate, SLA, failure-mode behaviour, owner. Integrations are where launches die.
  • Output · integration register, one row per system, with named owner
  • Trap · forgetting the "minor" integrations — analytics tags, CAPTCHAs, error reporters. They go down and they take you with them.
Caching topology — all five tiers
From the course's caching section. Browser · CDN · app-layer · origin (Redis) · data. For each tier, what lives there, what the invalidation strategy is, what TTL is used. Most production performance bugs originate from a gap in this diagram.
  • Output · caching diagram + invalidation runbook
Threat model — at architecture review
STRIDE pass on the architecture: Spoofing, Tampering, Repudiation, Information disclosure, Denial of service, Elevation of privilege. One bullet per attack surface, with the mitigation. Banking and regulated engagements need a formal threat model document.
  • Output · threat model document, signed off by security
Phase 4
Team shape & cadence
Week 4 · Output Team org + meeting cadence

Conway's Law applies before the first commit. The team shape you set up will be visible in the architecture six months later. Set it intentionally.

Squad structure — match the architecture you're building
Each squad owns a slice of the system end-to-end (front + back + ops), not a horizontal layer. Front-end-only squads create handoff hell at integration time. Squad sizes: 5–9 people including QA.
  • Output · squad chart, with each squad's domain boundary named
Meeting cadence — and the meetings you're killing
DSM (15 min, daily). Sprint planning (1h, biweekly). Retro (1h, biweekly). Architecture review (1h, weekly). Demo (30 min, end of sprint). Anything beyond this needs a justification. If it's already on the calendar without one, kill it.
  • Output · cadence calendar, named facilitators
  • Trap · "status meetings" with no decisions. Convert to async or kill.
Architecture office hours
90 minutes per week where any engineer can bring a design question, no agenda required. This is how you stay close to the code without hovering. The single highest-leverage time slot an architect has.
  • Output · weekly slot, async pre-submission
Phase 5
Foundations sprint
Week 5–7 · Output Walking skeleton

Before the first feature, ship a walking skeleton: minimal code, end-to-end, deployed to production. The skeleton proves the pipeline works, the architecture connects, and the team is unblocked. Skipping this phase costs you weeks later.

Pipeline first
CI from day one. Branch protection on. PR template. Linter + formatter + commit-message hook. Build that compiles. Deploy that lands a "hello world" in dev. The pipeline must exist before the first feature ticket is opened.
  • Output · green pipeline · "hello world" deployed
Observability wired up
Structured logs with correlation IDs. RED metrics emitted. Distributed tracing instrumented at the framework level. One end-to-end trace visible in the APM tool. Dashboards exist (even if empty).
  • Output · trace ID flowing end-to-end · RED dashboard live
Test scaffolding
Unit, integration, e2e directories with one example test each. Coverage reporting wired up. Coverage floor set in the pipeline. The team should be able to write the first real test without scaffolding work.
  • Output · test infra, one example test per layer, coverage floor enforced
Feature-flag infrastructure
Even if you don't need flags yet, install the SDK and the dashboard. Cost of installing later is much higher. Required if you're doing trunk-based.
  • Output · flag SDK installed, one boolean flag created and toggled in prod
Phase 6
First real feature
Week 7–9 · Output First customer-visible release

Pick the smallest valuable feature that exercises the full stack. Avoid the easy crud-screens trap; the first feature should test the architecture, not pretend it works. Hypercare like it's the final launch.

Hypercare plan
Define the hypercare window (typically 72 hours after first release). Named on-call. War-room channel pre-created. Rollback procedure rehearsed (not just documented). Business KPI dashboards visible to the team during the window.
Post-release retro — within 5 days
What broke that surprised us? What pipeline gap did we paper over? What process needs to change before the next feature? The first release surfaces every weak point in your setup. Fix them now.
Greenfield · the 90-day map
Day 0
Kickoff. You're named architect. Sign onto the channels, request access, get the org chart, read the brief.
Day 14
Discovery done. Checklist complete, stakeholder map drawn, constraints registered, named owners for every TBD.
Day 30
Architecture frozen v1. ADRs 001–004 signed. Topology + integration + caching diagrams done. Threat model presented to security.
Day 45
Walking skeleton live. Pipeline green, observability emitting, test scaffolding in place, feature flag toggled once in prod.
Day 60
First feature shipped. Hypercare window opened and closed. Retro done. First three lessons baked back into process.
Day 90
Team running. Sprint cadence steady. Architecture review weekly. Office hours used. Backlog has 3 sprints of clarity, not 12.
The greenfield trap: spending 30 days perfecting a design document before any code runs. The skeleton ships in week 5. The design will be wrong, and that's fine — running code surfaces the wrongness in days, not months.
Path B · Brownfield

Taking over an existing engagement

You've been parachuted into a system you didn't build, with a team you don't know, and a release calendar that doesn't pause for your onboarding. The principle: do no harm for 30 days. Ship, learn, refactor — in that order. The instinct is to start fixing immediately; resist it.

Phase 1
Code archaeology — first ten days
Day 1–10 · Output System map

Before you change anything, understand what's there. Most takeover mistakes happen in the first two weeks, from architects who diagnose the system through the lens of what they would have built.

Map the codebase using Stalwart layers
Even if the codebase doesn't use Stalwart, the layers are a useful lens. Walk the repo and tag each file: init, model, biz, api, render, state, events, config. The exercise surfaces what concerns are conflated and where the architectural seams are.
  • Output · annotated file map · concerns-mixed register
  • Trap · judging the existing structure. The map is for understanding, not grading.
Read every README — and notice what's missing
READMEs, architecture docs, runbooks, ADRs, post-mortems. Read them all. Then ask: what isn't documented? The undocumented operations (the "Joe always handles that" jobs) are where the risks live.
  • Output · documentation index · undocumented-operations register
Trace one real request end-to-end
Pick a representative URL. Follow it from DNS through CDN, WAF, LB, app server, every downstream call, every cache. Note what's instrumented and what isn't. You'll learn more about the system in this two-hour exercise than from a week of docs.
  • Output · sequence diagram of one real flow
Run a dependency & vendor audit
Every dependency: version, last update, security advisories, EOL status. Every vendor: contract end date, support SLA, who owns the relationship. Sometimes the biggest risk in a brownfield system is a library that lost its maintainer in 2022.
  • Output · dependency register · vendor register · risk-flagged
Phase 2
Stakeholder reintroduction
Day 5–15 · Output Updated stakeholder map

The previous architect built relationships you don't have. Rebuild them. Don't assume the political map you were handed is current — it's almost certainly outdated.

1:1 with every named stakeholder — within two weeks
Ask each person: what's working, what isn't, what would you change tomorrow if you could. Listen for what they don't say. The first 1:1 with each person is the most valuable; later they'll have learned what you want to hear.
  • Output · 1:1 notes per stakeholder · stance/influence map refreshed
Talk to one ex-engineer if you can
If a previous tech lead has left, talk to them. Off the record, no agenda. They know where the bodies are buried and they no longer have a reason to hide them. Bring a notebook.
Identify the "load-bearing humans"
Every brownfield system has one or two people without whom it falls over. Find them. They are simultaneously your most valuable asset and your most urgent risk. Plan succession on day one.
  • Output · bus-factor register · documentation backlog from each load-bearing human

Phase 4
Quick wins window
Day 20–40 · Output Visible improvements

The first 30 days establish credibility. Pick three high-pain / low-fix-cost wins from the matrix and ship them before day 40. Public, visible, measurable. This is what gives you political capital to spend on the harder fixes later.

Find the recurring incident
Read the last six months of post-mortems (or the incident channel, if there are no post-mortems). The same incident showing up monthly is your first quick win. Fix root cause, prove it stayed fixed.
Fix the trust-breaking metric
Every brownfield team has one number that's been bad for so long it's treated as immutable. Slow CI, flaky e2e, broken local dev setup. Fix it in your first 30 days and you earn the team's confidence faster than any architecture decision would.
Resist the rewrite urge
In every brownfield engagement there is a module that "obviously needs a rewrite." Almost always, it needs a refactor and not a rewrite. The rewrite trap: 6 months of work, 60% of original functionality, the same architectural sins reinstantiated. Refactor 80% of the time, rewrite 20%.
Phase 5
Re-cadence
Day 30–50 · Output Adjusted cadence

Once you understand the system, the team and the rhythm, adjust. Don't impose your preferred cadence on day one; you don't know enough yet. By day 40 you do.

Re-instate office hours
If they exist, attend. If they don't, start them. Same principle as greenfield, harder to install: the team needs to believe office hours are useful and that you won't use them as performance evaluation.
Architecture review — make it weekly
If the team has been shipping without architecture review, expect resistance. Frame it as decision documentation, not gatekeeping. ADRs first, opinions second.
Cut at least one meeting
There's always one. Status meetings that produce no decisions, standups that have grown to 45 minutes, post-launch reviews that nobody reads. Cutting a meeting is a credibility signal too — it shows you respect the team's time.
Phase 6
Roadmap re-baseline
Day 60–90 · Output Reset 6-month plan

By day 60 you've earned the right to question the existing roadmap. By day 90 you should have a re-baselined plan that the team owns — not one you imposed.

Strategic re-prioritisation
What's on the roadmap that no longer makes sense? What's not on the roadmap that should be? Bring the debt register from Phase 3 into the conversation explicitly. Trade business features against debt only with the customer's eyes open.
Ship the first roadmap item as the new architect
Same hypercare discipline as greenfield. Same post-release retro. This is the inflection: the moment you stop being "the new architect" and start being "the architect."
Brownfield · the 90-day map
Day 0
Handover. You take the seat. Don't change anything. Read.
Day 10
System mapped. Stalwart-layered file map. One full request traced. Documentation index built.
Day 15
1:1s done. Every named stakeholder met. Stance/influence map current. Load-bearing humans identified.
Day 25
Debt & risk registered. Matrix populated. Vendor + dependency audit done. Top three quick-win candidates identified.
Day 40
Quick wins shipped. Three high-pain / low-cost fixes visible. Trust-breaking metric improved.
Day 60
Cadence reset. Office hours live. Architecture review weekly. One meeting killed.
Day 90
Roadmap re-baselined. First roadmap item shipped under your watch. You're now "the architect."
The brownfield trap: starting with "this is all wrong" instead of "this exists for reasons I don't yet understand." Every odd-looking decision in a five-year-old codebase was rational at the time it was made. Your job is to find out what changed — then change what's worth changing.
Both paths

What's the same regardless

Greenfield and brownfield diverge in tempo and tactics, but the underlying discipline doesn't change. A handful of constants survive every engagement.

Always do

Decisions documented as ADRs

Every architectural choice has an ADR: the context, the decision, the alternatives, the consequences. Future-you and the next architect both depend on it. ADRs are 95% one-page documents; the discipline is in writing them at all.

Always do

Office hours, weekly

One 90-minute slot per week, no agenda. Engineers bring questions. You don't pre-prepare; you respond. The single highest-leverage hour an architect spends — and the one most often skipped because it feels unproductive.

Always do

Architecture review with teeth

Weekly. Time-boxed. Decision-oriented. Each item arrives with an ADR draft, leaves with an outcome (approved · revise · rejected). Without an outcome, the meeting is theatre.

Always do

API gateway before the integration list

The API gateway is the enforcement layer for every integration: rate limiting, auth, logging, versioning, and routing all live there. Decide the gateway — AWS API Gateway, Kong, Azure APIM, Apigee — before any integration is built, not after the third integration breaks prod. The rule is simple: if an external system calls your API, or your API calls an external system, it goes through the gateway. Internal service-to-service calls are a separate ADR.

Always do

Auth protocol decision in Discovery, not in Sprint 3

SAML vs OIDC vs OAuth 2.0 is not a developer decision. The architect decides in Discovery, with the security team and the IdP owner — a wrong choice in Sprint 3 is a rewrite in Sprint 7. The heuristic: SAML for enterprise SSO into legacy systems; OIDC for modern web and mobile; OAuth 2.0 client credentials for machine-to-machine; JWT as the token format for stateless APIs. They compose — OIDC sits on top of OAuth 2.0; SAML and OIDC can be bridged via an identity broker. See the Architect's FAQ for regulated-industry auth patterns.

Always avoid

Bypassing the API gateway for "just one integration"

Every bypassed integration becomes a shadow integration: no rate limiting, no auth enforcement, no logging, no circuit breaker. The first bypass is a precedent; the tenth is a security incident. If the gateway adds friction, fix the gateway — don't route around it.

Always avoid

JWT as a session mechanism without rotation

JWTs are stateless and cannot be revoked once issued — a stolen token with a 24-hour expiry is valid for 24 hours with no remediation path. The correct pattern: short-lived access token (15 minutes) paired with a long-lived refresh token that rotates on every use, with the previous token immediately invalidated. Session termination invalidates the refresh token server-side; the access token simply expires.

Always avoid

The hero pattern

When the architect becomes the bottleneck on every decision, the team stops growing. Push decisions down. The right level for a decision is the lowest level that can be informed about it; the architect's job is making sure that level has the information.

Always avoid

The rewrite pitch in month one

It's tempting and almost always wrong. You don't yet understand the constraints that produced the current state. The Chesterton's Fence rule: never tear down a fence until you know why it was put up.

Always avoid

Standalone architecture meetings

Architecture decisions made by architects alone, presented to the team as fait accompli. Decisions are durable only when the people implementing them helped shape them.

The runbook isn't a checklist. The phases are sequenced for a reason — Phase 2 needs Phase 1's discovery; Phase 5 needs Phase 4's cadence. Skipping ahead is a common failure mode. So is staying too long in any one phase. The 90-day map exists so you have a forcing function to move on, even when the work feels unfinished. Architecture work is always unfinished; that's not a reason to delay the next phase.
An architect's first 90 days look the same shape regardless of project. Discover, decide, design, build the team, build the skeleton, ship. The greenfield version moves faster; the brownfield version moves more carefully. Both end in the same place — first release under your watch, hypercare opened and closed, the second feature in the backlog.
