# Content Author Guide

How to add or change each kind of content in DEPT® Anatomy of Code. Five content
types, two models:

- **Dynamic** (DB → API → app): the **course** and **media**. Edited via files/CMS
  or uploaded through the app; served live.
- **Static** (plain HTML, served by Apache): **FAQs, checklists, runbooks**. Each
  lives under `resources/<section>/` as a folder with an `index.html` landing and
  one HTML page per item. No API, no database — reliable by design.

The modular apps — **Quiz** and **Techflix** — are not covered here; they manage
their own content (quiz bank, Techflix episodes via `techflix.json`).

> House voice (all content): Indian English spelling (organise, optimise,
> behaviour), plain professional register, acronyms expanded on first use. See
> `CLAUDE.md`.

## Who can do what (permissions)

| Action | Permission | Held by |
|---|---|---|
| Upload media via the app | `media.upload` | `content_author`, `feed_contributor` |
| Upload/publish runbooks via the API | `content.write` | `content_author` |
| Edit static pages (FAQ / checklist / runbook HTML) | — (git/file access) | repo committers |
| Edit course source + re-publish | — (git/file access) | repo committers |

`platform_admin` bypasses all checks. Roles are managed in Directus and synced to
the app; see `docs/architecture/v2/04-authz-model.md`.

## 1 · Main app (the course)

The course shown in the SPA (Manual / Read modes) is data-driven.

- **Source of truth:** the git-tracked JSON in `content/source/course/` —
  `framework.json`, `framework-explainer.json`, and one file per chapter under
  `content/source/course/sections/*.json`.
- **Runtime:** an ETL loads those JSON files into Postgres (`frameworks`,
  `course_chapters`); the SPA reads them via `GET /api/course/*`.

### To add or edit a chapter
1. Edit (or add) the chapter JSON in `content/source/course/sections/`. Follow the
   **LAYER pattern** (`CLAUDE.md`): a Scan Box at the top of each major section,
   the declarative prose underneath (don't bulletise the prose), diagrams/callouts
   woven through. Keep the four callout types (Why This Matters / Agency Tip /
   Common Pitfall / Before-After).
2. If the chapter is new, add its filename to `SECTION_FILES` in
   `frontend/core/config.js` so Contents and the Manual know it exists.
3. Publish to the database:
   ```bash
   cd backend && .venv/bin/python -m scripts.migrate_to_postgres
   ```
4. The SPA picks it up on next load (reads are cached ~15 min, or invalidated
   immediately when edited through Directus).

> New section / deepblock / architectural concept? The quiz curator (`q0`) should
> add question-bank coverage, and `content-quality` should review brand/voice/
> AI-tell discipline (project rule).

## 2 · FAQs (static)

Lives in `resources/faqs/`:
```
resources/faqs/
├── index.html             ← the FAQs landing (cards → each FAQ page)
└── aem-banking-faq.html   ← one FAQ page (a zero-JS <details> accordion)
```

### To add a question to an existing FAQ
Edit the FAQ page (e.g. `resources/faqs/aem-banking-faq.html`) and add one block:
```html
<details class="faq">
  <summary><span class="qn">14</span><span class="qt">Your question here?</span><span class="plus">+</span></summary>
  <div class="faq-a">
    <p>The answer, in DEPT voice. <strong>Bold the key point.</strong></p>
    <h5>A sub-heading if useful</h5>
    <div class="chips"><span class="chip">Network</span><span class="chip">Security</span></div>
  </div>
</details>
```

### To add a whole new FAQ topic
1. Copy `aem-banking-faq.html` to `resources/faqs/<topic>.html`, change the title +
   questions.
2. Add a card to `resources/faqs/index.html` pointing at it.

Served at `/resources/faqs/`. No build, no API, no database — the static HTML is
the single source of truth.

## 3 · Runbooks (Excel template → static)

Authored from the **Excel template** and published to a static page.

### Steps
1. **Get the template:** `backend/data/runbook-template.xlsx` (generated by the
   deploy, or run `python -m scripts.generate_runbook_template <path>`).
2. **Fill it.** Two sheets:
   - **`Runbook`** — `key | value` rows: `Slug`, `Title`, `Role`
     (architect/devops/developer/qa/pm/ba), `Domain` (banking/ecommerce/…),
     `Type` (greenfield/brownfield), `Status` (draft/published), `Description`.
   - **`Content`** — header `type | title | description | owner | tools | timing | url | notes`,
     then one row per item where `type` is `phase`, `section`, `task`, `step`,
     `checklist`, or `link` (rows nest under the most recent phase → section → task).
3. **Publish to a static page:**
   ```bash
   cd backend && .venv/bin/python -m scripts.render_runbook /path/to/filled.xlsx
   ```
   This writes `resources/runbooks/<slug>.html` and rebuilds the landing
   `resources/runbooks/index.html` (a card per runbook). Served at
   `/resources/runbooks/`; short URL `/runbook` redirects there.

Re-running with the same slug overwrites the page (idempotent). To remove one,
delete `resources/runbooks/<slug>.html` and re-run any `render_runbook` (the index
rebuilds from the files present).

## 4 · Checklists (static)

Lives in `resources/checklists/`:
```
resources/checklists/
├── index.html                   ← landing
└── code-coder-checklist.html    ← the checklist
```
Edit the checklist HTML directly; to add a new checklist, copy the file to
`resources/checklists/<name>.html` and add a card to `index.html`. Served at
`/resources/checklists/`.

## 5 · Media (videos & images)

All media lives in Postgres large objects under the unified video model
(`video_asset` → `video_variant` → files); see `docs/architecture/v2/09-video-model.md`.
Three author paths, by where the video belongs:

### a) Feed video — from the app UI (content authors)
In the feed composer, pick **Video** → choose a file. It uploads via
`POST /api/media/upload` (needs `media.upload`), and the post streams it. Nothing
else to do.

### b) Techflix episodes — `techflix.json` + CLI
Drop the `.mp4`s + a `techflix.json` in a folder, then on the server:
```bash
cd backend && .venv/bin/python -m scripts.media techflix /path/to/folder
```
Full details + the manifest format: `TECHFLIX.md`.

### c) General / course-embedded video — CLI + a slug
For a video you want to embed anywhere (e.g. the course explainer):
```bash
cd backend
.venv/bin/python -m scripts.media upload /path/to/folder        # ingest
.venv/bin/python -m scripts.media list                          # find the new id
.venv/bin/python -m scripts.media set-slug <video_asset_id> explainer
```
Then embed it with the stable slug: `/media/video/explainer` (no per-environment
UUIDs). Manage with `scripts.media list | set-slug | rm`. Full reference: `MEDIA.md`.

> The Directus admin also has a media-upload screen (a thin client of
> `POST /api/media/upload`). The `scripts.media` CLI runs server-side as an
> operator and bypasses the permission check — it's for maintainers, not
> self-serve content authors.

## Quick reference

| Content | Where | How to publish |
|---|---|---|
| Course | `content/source/course/*.json` | `scripts.migrate_to_postgres` |
| FAQs | `resources/faqs/*.html` | commit (static) |
| Runbooks | Excel template | `scripts.render_runbook <file.xlsx>` → static |
| Checklists | `resources/checklists/*.html` | commit (static) |
| Feed video | app UI (Video) | `POST /api/media/upload` |
| Techflix | `techflix.json` + folder | `scripts.media techflix <folder>` |
| Embedded video | folder | `scripts.media upload` + `set-slug` |
