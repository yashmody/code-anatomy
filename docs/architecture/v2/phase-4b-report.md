## 1. Gate decision

**GO for the 4b gate.**

Phase 4b unified the three same-origin surfaces — the SPA (`/app`), the frozen
resource islands (`/anatomy`) and the quiz (`/`) — onto a single theme key and a
consistent header, and stood up the frontend moderator UI over the moderation

  `git diff c67e211..v2` is empty for `content/source/` (all course JSON),

  `content/frozen` changes are the resource pages 4b is explicitly permitted to
  edit, plus two new helper files (`theme-boot.js`, `_header.html-snippet`).
- **Smoke 15/15** throughout, including the strict `SMOKE_REAL_CERT_CHECK=1` run
  where the cert canary `CCA-F-20260605-E79E74AB` returns `valid=true`
  (size 11914). The new negative checks (`/auth/me` → 401,
  `/api/moderate/queue` → 401) pass. FastAPI `from app.main import app` imports
  clean; `bash -n deploy.sh` is clean.
- **Schema is untouched.** Alembic head stays `0008`. 4b is additive over the
  API and the frontend only — no migration, no DDL. That is by design: the

## 2. Per-slice summaries

### Slice 4b-A — Backend: roles+permissions, moderation cache, SSO reconcile

Commit `5d07132` (4 files, +77/-9): `backend/app/modules/auth/routes.py`,
`backend/app/modules/feed/routes.py`, `deploy.sh`, `cms/.env.example`.

- **`/auth/me` is additive.** A module-level helper `_permissions_for_roles`
  (auth/routes.py:187) reverse-maps `core.deps.PERMISSION_GRANTS` so a user holds
  a permission iff their role set intersects the grant set, with `platform_admin`
  returning the full matrix (mirroring `require_permission`'s bypass). Both
  `roles` and the new `permissions` derive from a single `roles_for(email)` read.
  Every pre-existing field (`userId, email, name, picture, persona, provider,
  preferences, initials, roles`) is intact; unauthenticated behaviour stays 401.
- **Cache invalidation** added to `moderate_action`: after the status flip commits,
  a feed action drops `feed_items:` (feed/routes.py:133) and a question action
  drops `questions:` (feed/routes.py:149) — type-scoped, matching the Directus
  webhook prefixes in `cms/routes.py`. Moderation writes land in the FastAPI plane
  and bypass the Directus webhook, so they must self-invalidate to keep
  feed-list / queue / quiz-pool reads fresh.
- **SSO reconciled** (secure posture, `deploy.sh` ⇄ `cms/.env.example` agree):
  `AUTH_GOOGLE_ALLOW_PUBLIC_REGISTRATION=false`,
  `AUTH_GOOGLE_ALLOW_LIST=deptagency.com`, `AUTH_GOOGLE_DEFAULT_ROLE_ID`
  (least-privilege `content_author` placeholder). Self-registration is off (admins
  pre-create staff users matched to their @deptagency.com Google email); the
  allow-list is defence-in-depth domain enforcement; the default role is inert
  while registration is false but set least-privilege so a future flip stays safe.

### Slice 4b-C — Nav shell + role-gated moderator view

Commit `4d1f39e` (5 files, +563/-6): `frontend/index.html`,
`frontend/core/main.js`, `frontend/core/auth-ui.js`, new
`frontend/modules/moderate/moderate.js`, new `frontend/styles/moderate.css`.

- **Nav (`index.html`):** the three `/anatomy/*` Resources links are now same-tab
  (`target=_blank rel=noopener` dropped); 4b-B's shared back-header carries the
  reader back. The Quiz link **keeps** `target=_blank` (separate page set). A
  `Moderation` mode-tab (`#navModerate`, `href="#/moderate"`) is present but
  `hidden` by default.
- **Permission gating (`auth-ui.js`):** `permissions[]` rides on the `/auth/me`
  session returned by `getCurrentSession()`. `syncModerationNav()` unhides
  `#navModerate` iff `permissions` includes `moderate.view`, and runs on boot and
  after every sign-in/sign-out. The server-side `require_permission` gate is
  authoritative regardless.
- **Router (`main.js`):** `#/moderate` shows a "Not authorised" placeholder (no
  fetch) when `hasPermission('moderate.view')` is false; otherwise it lazy-imports
  `modules/moderate/moderate.js`. Lazy-loading keeps the moderator route off the
  static import graph.
- **Moderator view (`moderate.js`):** `GET /api/moderate/queue` renders two
  sections — flagged feed items (reusing the feed card body) and questions
  (compact review cards with the correct answer marked). Per-item Approve / Flag /
  Remove → `POST /api/moderate/action` with `{item_id, item_type, action}`,
  matching `ModActionPayload` exactly. Remove uses an accessible inline confirm
  (real buttons, ESC/Cancel, focus restore); 401/403/network handled; "Queue is
  clear" empty state. It inlines the correct `renderDiagram` import from
  `../../shared/render/diagram.js` to route around the pre-existing broken
  `feed/media.js` import (see §7).

## 8. What unlocks 4c

4b deliberately left the schema untouched (Alembic head still `0008`). 4c is the
final, schema-mutating slice: **live authoring + relational course decomposition**.

- **Relational `course_chapters` decomposition.** Today the course renders from
  the frozen JSON / monolith. 4c decomposes the course into relational tables
  (chapters and their ordered content), with an Alembic migration past `0008`,
  so Directus can author chapters directly rather than the course living as a
  frozen artifact. This is the last byte-identical guarantee to fall — by design,
  and only in 4c.
- **Live authoring path.** The Directus editorial plane (phase 4a) and the
  moderation/permissions contract (4b-A) are the substrate; 4c wires content edits
  through to the relational store and the FastAPI read path with cache
  invalidation already in place from 4a/4b.
- **Prerequisite hygiene before 4c.** The pre-existing broken feed/course imports
  (`task_be3e627d`) should be fixed first, so the SPA Manual/Read/Feed surfaces
  boot in-browser and 4c authoring can be verified end-to-end against a live SPA,
  not only structurally.

4c is the last slice of the v2 re-architecture.
