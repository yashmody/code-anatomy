Architect-level answers to the questions that come up in every regulated-industry AEM implementation. Written for architects, with security teams as the secondary audience. Search the questions, filter by tag, or browse all thirteen.

01Traffic from the Bank's Akamai CDN to Adobe-managed DNS must use a dedicated, whitelisted route, not the public internet. Is this the case?+

The short answer: no, not in the strict private-peering sense. Adobe Managed Services does not provide dedicated private circuits (such as ExpressRoute or Direct Connect) between a customer's CDN provider (Akamai) and AMS infrastructure.

The standard pattern

Traffic flows over the public internet using HTTPS (TLS 1.2+), secured by two complementary controls:

  • IP allow-listing — Adobe MS ingress firewall is configured to accept traffic only from Akamai's published egress IP ranges. These ranges update periodically; track them as an operational dependency.
  • Origin authentication — Akamai signs requests with shared secrets (origin cloaking via signed headers), or mTLS where contractually supported.
For higher assurance

mTLS between Akamai and Adobe origins can be negotiated as a contract addition. Some banking customers also deploy a reverse proxy in their own DMZ between Akamai and AMS to add a tenant-controlled inspection point.

For the architectThe defensible answer to cyber security is: TLS-over-internet, IP-restricted on both ends, origin-authenticated, optionally mTLS. Document the Akamai IP allowlist owner and the renewal cadence — this is the most common operational gap.

02The architecture diagram must explicitly show Adobe's own WAF/CDN layer in addition to the Bank's Akamai layer. The cyber-security team needs visibility into the full defense-in-depth chain.+

The defense-in-depth chain for AEM AMS behind Akamai has six labelled layers. Every architecture diagram for a banking review should show all of them with vendor, role, owner, and the security control each enforces.

The chain, in order
  • 1 · Akamai (Bank-owned) — first-tier CDN + WAF. Absorbs DDoS, edge caches, bot protection, IP reputation filtering.
  • 2 · Adobe ingress LB — TLS termination at the AMS edge for traffic from Akamai.
  • 3 · Adobe WAF — AMS includes a managed WAF (AWS WAF or equivalent depending on tenant). Rule sets are jointly managed with Adobe and tunable per environment.
  • 4 · AEM Dispatcher — application-layer cache and URL filter. The AEM-native control for path, method, and parameter-based denials.
  • 5 · AEM Publishers — read-only customer-facing instances, segmented behind the dispatcher.
  • 6 · AEM Author — content authoring, never internet-facing. Accessed via VPN or bastion only.
For the architectCyber security teams want a kill-chain mapping — which layer stops which attack class (volumetric DDoS at Akamai, application-layer at Adobe WAF, path-based at Dispatcher). Build the diagram with these annotations and a column showing "owned by Bank" vs "owned by Adobe" for each layer.

03How is the AEM MS physically and logically separated from other tenants on the same Adobe Managed Services infrastructure?+

AEM Managed Services is not shared-tenancy SaaS — it is a managed dedicated infrastructure model. Each customer runs on isolated cloud infrastructure provisioned and operated by Adobe.

Isolation guarantees
  • Dedicated VPC per customer on the underlying cloud (AWS or Azure depending on tenant).
  • Dedicated compute — Author, Publisher, and Dispatcher VMs are not shared across customers.
  • Dedicated network — subnets, security groups, route tables, NSGs are tenant-specific.
  • Dedicated storage — disk volumes, snapshots, and object storage (S3 or Blob) are tenant-scoped with KMS encryption.
  • Shared services — only the Adobe Cloud Manager control plane and Adobe IMS identity backbone are shared infrastructure; these are orchestration/identity layers, not data planes.
Applicable compliance certifications

SOC 2 Type II · ISO 27001 · PCI DSS · HIPAA (tier-dependent). Banking customers should request the most recent attestation reports and the AMS tenant-isolation architecture document.

For the architectGet the tenant-isolation documentation specific to your contract from Adobe — particularly the VPC/subnet topology, KMS key boundaries, and the list of Adobe operators with access (usually time-bounded and just-in-time).

04The target throughput is 500 TPS. The team was asked to elaborate on the auto-scaling mechanism and to derive a capacity view for performance testing.+

AMS supports horizontal scaling of Publisher and Dispatcher tiers. Author is typically a single primary with a hot standby. Auto-scaling is not as elastic as cloud-native services — scale events are managed by Adobe operations.

Capacity model at 500 TPS
  • Cache hit ratio is the lever — at 90% dispatcher cache hit, only 50 TPS reaches Publishers. At 50%, 250 TPS hits Publishers. Plan against the worst plausible cache miss scenario (campaign launch + cold cache + cache invalidation event).
  • Per-Publisher throughput — rule of thumb is 50–80 RPS sustained at moderate page complexity. Always validate with synthetic load tests; never assume.
  • Auto-scaling reality — AMS pre-provisions peak capacity based on agreed SLA. Adding instances mid-campaign typically requires a ticket and lead time. Plan for peak, not for elasticity.
Defensible starting capacity for 500 TPS
  • 4–6 active Publishers (page mix dependent)
  • 2 Dispatchers in active-active with shared cache strategy
  • Author + hot standby
  • Reserved headroom of 30% for unplanned bursts
Mandatory pre-go-live tests
  • Synthetic load at 1.5× target TPS — so 750 TPS sustained.
  • Burn-in test at 500 TPS for at least 2 hours.
  • Cold cache scenarios at 500 TPS.
  • Cache invalidation storm test.
For the architectThe capacity number Adobe quotes is not the number you should plan against. Build your capacity view from cache hit ratio, page mix, and realistic peak modelling — then validate with load tests, twice, before production cutover.

05Where exactly is Adobe Cloud Manager hosted, and does its operation stay within the tenant boundary assigned to the client?+

Adobe Cloud Manager is Adobe's CI/CD orchestration platform, hosted as multi-tenant SaaS on Adobe's own cloud infrastructure. It is operationally separate from the customer's AMS tenant but acts on it.

What sits where
  • Cloud Manager UI, API, pipelines, metadata — hosted by Adobe globally (multi-tenant).
  • Source code repository — Adobe-hosted Git, with optional mirroring to the customer's own Git.
  • Deployment targets (your AEM Author, Publishers, Dispatchers) — your dedicated AMS tenant.
  • Authentication — Adobe IMS, federated to your IdP (Azure AD / Entra) for Single Sign-On.
  • Deployment execution — Cloud Manager initiates actions in your tenant via secure Adobe-managed control-plane connections.
Implications for a regulated environment
  • Confirm with Adobe which region(s) host Cloud Manager metadata for your tenant.
  • Verify Cloud Manager audit logs are exportable to your SIEM (Splunk).
  • Federate Cloud Manager users to the Bank's IdP via Adobe IMS — no local accounts.
  • Code does not "leave" the Adobe boundary; IMS controls who can trigger deployments.
For the architectThe control plane is shared, the data plane is not. For most banking compliance reviews, this distinction is acceptable provided IMS federation and audit log export are in place. Document both.

06What encryption is applied to data at rest? How are encryption keys managed? Is Bring Your Own Key (BYOK) supported?+

Default AMS encryption posture is strong on the basics and conservative on BYOK. Banking requirements often hit the BYOK question hard — surface this early.

Default encryption
  • Data at rest — all disk volumes encrypted using the underlying cloud's native KMS (AWS KMS for AWS-hosted, Azure Key Vault for Azure-hosted).
  • Data in transit — TLS 1.2+ between all components, internal and external.
  • AEM JCR data — encrypted at the disk level (not field-level).
  • Backups — encrypted with the same KMS keys; snapshots inherit volume encryption.
Key management — default
  • Adobe-managed keys — Adobe controls KMS for your tenant by default.
  • Key rotation per Adobe's standard policy.
  • Key access restricted to Adobe operators with just-in-time access.
BYOK status
  • AMS has had limited or no BYOK historically — needs explicit contract confirmation.
  • Recent AMS configurations support customer-managed keys via cross-account KMS arrangements in some cases; contract-specific.
  • AEM as a Cloud Service has broader BYOK support than AMS — worth comparing if you're choosing between deployment models.
For the architectGet BYOK status in writing from Adobe before signing. If BYOK is a hard compliance ask and AMS doesn't support it for your contract, the compensating controls are: data classification, tenant isolation, KMS audit logging, and time-bounded operator access. Document these explicitly.

07What is the backup and archival strategy? Where are backups stored? How does this align with the Bank's RTO/RPO targets?+

AMS's standard backup model is daily snapshots with 30-day retention. Banking RTO/RPO targets often exceed the standard tier — escalate before contract signature.

Standard backup model
  • Daily snapshots of all production VMs and storage (Author, Publishers, Dispatchers).
  • Default retention — 30 days. Extended retention (90 days, 1 year) is contract-configurable.
  • Cross-region replication — optional, tier-dependent.
  • Storage — Adobe-managed encrypted storage in the same cloud region (and replica region if DR is enabled).
Standard RTO / RPO by tier
  • Standard tier — RPO 24h · RTO 8h+
  • Gold tier — RPO 24h (configurable down to ~1h with transaction log shipping) · RTO ~4h
  • Platinum tier — RPO ~1h · RTO ~1h
Banking-specific considerations
  • If your RTO is 1h and RPO is 15min, standard AMS configurations will not suffice — discuss DR architecture with Adobe in detail.
  • Document cold-DR vs warm-DR cost trade-off explicitly with finance.
  • Backup restoration testing should be a contract clause — twice yearly minimum, with a written test report.
  • For regulated data (transaction records), consider WORM archival outside AMS.
For the architectThe RTO/RPO numbers Adobe quotes are the platform's; your business may have stricter targets for specific data domains. Map each data domain to its acceptable RTO/RPO and validate the tier purchased actually meets the strictest of these. This is a frequent miss in contract review.

08Is Splunk part of the client's tenant? If yes, it should be shown in the architecture diagram with the data flow clearly depicted.+

Splunk lives in the Bank's tenant, not in Adobe's. AMS does not include Splunk; integration is via log shipping.

Integration patterns
  • Splunk HEC (HTTP Event Collector) — most common. AMS forwards logs over HTTPS to your Splunk endpoint with token authentication.
  • Syslog forwarder — push to a syslog collector that feeds Splunk.
  • API pull — Splunk pulls from Adobe's log export endpoint; less common, more brittle.
Logs typically available from AMS
  • AEM application logs (error.log, request.log, access.log)
  • Dispatcher logs
  • Web server access logs
  • OS-level logs (limited; Adobe controls VM access)
Architecture diagram should show
  • AMS log sources (Author, Publishers, Dispatchers) →
  • Adobe log aggregation tier →
  • HTTPS forwarder over allowlisted egress →
  • Bank's Splunk HEC endpoint (in Bank's tenant) →
  • Splunk indexers, search heads, and dashboards.
For the architectSplunk HEC tokens should be rotated quarterly and stored in the Bank's secrets vault, not in AMS configs. Estimate ingestion volume early — a busy AEM site can produce 5–20 GB/day of access logs. Confirm with Adobe which operator-action logs are exportable vs withheld; this is often a compliance discussion point.

09Bank has Dynatrace. Is integration with Dynatrace with AEM MS feasible?+

Yes, integration is feasible and commonly done. Three integration paths exist; the right choice depends on what depth of visibility the Bank needs.

Integration options
  • Dynatrace OneAgent installation on AMS VMs — gold standard. Requires Adobe approval and is treated as a managed software install. Adobe handles deployment and updates per release cycles. Provides full-stack APM: JVM metrics, GC, database calls, transaction tracing.
  • Dynatrace synthetic monitoring — Bank-side, no Adobe involvement. External probes hit AEM endpoints on a schedule. Good for availability and synthetic transaction monitoring.
  • Dynatrace API integration — pull metrics via Adobe-exposed APIs into Dynatrace as a data source. Less rich than OneAgent.
Practical notes
  • OneAgent install is contract-dependent — confirm scope with Adobe before assuming it's available.
  • Once installed, the Bank's Dynatrace tenant ingests data directly from AMS instances; Adobe ops do not see your Dynatrace data.
  • Performance overhead is minimal — typically 1–3% CPU.
  • Dynatrace OneAgent licensing is the Bank's responsibility, not Adobe's.
For the architectIf the Bank already runs Dynatrace as its enterprise APM, push for OneAgent installation on AMS — it gives the same visibility you get on your own infrastructure. If Adobe resists, ask for the specific contractual or operational reason; the answer is usually "not in current scope" rather than a hard no.

10The Publisher-to-VBCS (Oracle Cloud) integration needs elaboration. API calls must not be anonymous and should be authenticated, authorized, and auditable.+

AEM Publisher calling Oracle VBCS (Visual Builder Cloud Service) APIs must use authenticated, authorised, and audited channels. Anonymous calls are not acceptable for a banking integration.

Recommended pattern
  • OAuth 2.0 with IDCS (Oracle Identity Cloud Service) — IDCS issues access tokens to AEM via the client credentials grant. AEM presents tokens to VBCS APIs.
  • Token caching — AEM caches tokens for their lifetime (typically 1 hour) to avoid token-acquisition round-trips on every request.
  • mTLS (recommended for banking) — additional certificate-based authentication on top of OAuth.
  • JWT assertions as an alternative grant type for service-to-service.
What must be in place
  • A dedicated service account in IDCS for AEM → VBCS calls (not a user account).
  • API audit logging on both sides — VBCS records every authenticated request; AEM logs outbound calls with correlation IDs.
  • Rate limiting in VBCS per service account.
  • Secret rotation — service account credentials rotated quarterly minimum; tracked in your secrets vault.
  • Network controls — outbound traffic from AEM Publishers to VBCS traverses the Bank's network controls (proxy, allowlist).
What must be documented
  • Full request flow including token acquisition, refresh logic, and failure modes.
  • Graceful degradation when VBCS is unreachable — no error pages to end users.
  • Retry policy with exponential backoff and circuit breaker.
  • End-to-end trace correlation between AEM and VBCS for incident response.
For the architectThe integration spec should be a separate document signed off by both AEM and Oracle teams before development starts. Include sample code for token acquisition, a deployment runbook for credential rotation, and a test plan covering token expiry, rate limiting, and VBCS unavailability.

11Segment the Application Gateway, Dispatcher, and Publisher into distinct network zones.+

Standard AMS network architecture already segments tiers into separate subnets. The architect's job is to verify the segmentation matches the Bank's reference architecture and to audit the firewall rules.

Recommended segmentation
  • DMZ subnet — Application Gateway / Load Balancer. Internet-facing (behind Akamai).
  • Web tier subnet — Dispatchers. Receive traffic from AppGW only; outbound to Publishers only.
  • App tier subnet — Publishers. Receive traffic from Dispatchers only.
  • Secure tier subnet — Author. Receives traffic from VPN or bastion only; pushes content to Publishers via replication.
  • Data tier subnet — object stores and backup volumes. Accessed via IAM-controlled paths only.
Required controls
  • Network Security Groups (NSGs) / Security Groups deny-all by default; explicit allow rules per port and source.
  • East-west traffic restricted between tiers — no Publisher-to-Publisher direct calls, no skip-tier paths.
  • No direct internet egress from Publisher or Author tiers, except whitelisted destinations (VBCS, Azure OpenAI, etc.).
  • All inter-tier traffic logged and exported to SIEM.
For the architectRequest the AMS network architecture diagram from Adobe. Verify NSG rules against the Bank's reference architecture line-by-line. Audit egress allowlists — these are the most commonly-missed control and the most exploitable. Each whitelisted destination should have a documented owner and a quarterly review.

12How will Digital Personal Data Protection (DPDP) Act opt-outs will get handled?+

India's DPDP Act 2023 mandates clear opt-out and erasure mechanisms for personal data processing. For an AEM implementation, the architecture must support consent capture, propagation, suppression, and audit — across every system that touches user data.

Required architectural capabilities
  • Consent capture at point of collection — every form, cookie banner, and user touchpoint records consent state with timestamp and version.
  • Centralised consent store — single source of truth per user, typically a Consent Management Platform (OneTrust, TrustArc) or the Bank's identity service.
  • Propagation — opt-out signals must reach AEM, Adobe Analytics, Adobe Target, CDP, CRM, email systems, and ad platforms within a defined latency budget (minutes, not days).
  • Suppression in AEM — personalization rules in AEM Sites / Target must check consent state before activating.
  • Right to erasure — when a user requests deletion, every system purges identifiable data; AEM content fragments referencing user data must be regenerated.
  • Audit trail — every consent change, opt-out, and erasure logged with timestamps and operator.
Latency and reliability
  • Define a consent propagation SLA — for example, "all systems reflect opt-out within 15 minutes of capture."
  • Monitor the propagation pipeline; alarm on lag.
  • Build a consent kill switch — an emergency mechanism to suspend personalization globally if the consent system fails.
For the architectMap every data collection point in the Bank's digital estate to its consent capture mechanism and downstream propagation path. Gaps in the map are gaps in DPDP compliance. The CMP is not enough on its own — the propagation pipeline and downstream enforcement need equal architectural rigour.

13AI-enabled search capabilities via Azure OpenAI integration were discussed. Concerns were raised around rate limiting, request management, and cost governance to prevent unexpected usage spikes.+

Azure OpenAI in a banking AEM context requires governance at three layers: Azure service limits, application-level controls, and cost management. Treat AI cost like cloud cost — without governance, it grows without bound.

1 · Service-level limits (Azure side)
  • Azure OpenAI deployments have TPM (tokens per minute) and RPM (requests per minute) quotas configured per model deployment.
  • For production, request Provisioned Throughput Units (PTU) instead of pay-as-you-go to avoid surprise throttling and get predictable cost.
  • Configure quota alerts at 70% and 90% utilisation.
2 · Application-level controls (between AEM and Azure OpenAI)
  • API Gateway in front of Azure OpenAI calls — adds a controlled enforcement point for rate limiting per user, per session, per IP.
  • Per-user token budgets — cap how many tokens an individual session can consume.
  • Prompt length caps — reject prompts above N tokens before they hit Azure.
  • Semantic cache — cache common queries using embeddings to avoid repeat costs (often 30–50% cost reduction for FAQ-style queries).
  • Fallback patterns — degrade gracefully to keyword search or static content when AI is unavailable or rate-limited.
3 · Cost governance
  • Budgets and alerts in Azure Cost Management — daily, weekly, monthly thresholds with email and Slack alerts at 50%, 80%, 100%.
  • Per-prompt cost logging — log every request's token count and cost for chargeback.
  • Dashboard showing real-time spend, projected monthly spend, top users and queries by cost.
  • Auto-disable kill switch — automated suspension if daily spend exceeds threshold, requiring manual re-enable.
Banking-specific guardrails
  • PII detection on prompts — block requests containing customer-identifying data before they leave the Bank's perimeter.
  • Output filtering — block responses that might leak training data, internal information, or hallucinated financial advice.
  • Audit logging — every request: who, what prompt, what response, when. Retain per DPDP and Banking record-keeping rules.
  • Disclosure — be explicit to users that responses are AI-generated.
For the architectThe biggest risk with Azure OpenAI in production is not technical — it's cost. A single misconfigured loop, scraper, or runaway script can burn six figures in a weekend. Treat the API Gateway, per-user budgets, and kill switch as non-negotiable, not as future enhancements.
