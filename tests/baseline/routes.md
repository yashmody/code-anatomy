## Auth model (current)

Two enforcement primitives exist in the v1 monolith:

1. **Session redirect** — `_require_user` / `_require_user_with_role`
   (`main.py:79-92`). No session → raises `HTTPException(302, Location=/login)`.
   No role on the session user → raises `HTTPException(302, Location=/onboarding/role)`.
   These are page-style guards; they redirect the browser.

2. **Capability check** — `auth.require_role(allowed: List[str])` dependency
   (`auth.py:107-136`). Accepts capability strings: **`User`**, **`FeedCreator`**,
   **`Moderator`**, **`QuizManager`**. `QuizManager` bypasses every other check
   (`auth.py:128-130`). On API/JSON paths it returns `401`; otherwise it 302s to
   `/login`. On wrong role it returns `403 Operation forbidden`.

The `users.role` column is currently **overloaded**: it stores either a
**persona** key from `roles.py` (`pm`, `ba`, `qa`, `sales`, `design`, `devops`,
`coder`, `architect`, `other`) **or** a capability string (`User`, `FeedCreator`,
`Moderator`, `QuizManager`). The two systems are not formally mapped — see
[`04-authz-model.md` §1.3](../../docs/architecture/v2/04-authz-model.md). Phase
2b splits this into a separate `user_roles` table; the baseline below documents
the v1 behaviour exactly as observed.

DEV_MODE side-effect: `storage.upsert_user` auto-assigns `role="QuizManager"` to
brand-new users when `DEV_MODE=true` (`storage.py:74`). This is why the dev
smoke tests can hit moderator/admin paths without explicit role assignment.

## Parity guarantees Phase ≥1 must preserve

For every row above, the post-restructure backend must keep:

1. **Path string identical.** `/api/course/framework-explainer`, not
   `/api/course/explainer`, etc. (Front-end fetches are hard-coded — see
   `01-blueprint.md` §7 "Front-end fetch endpoints + constants".)
2. **HTTP method identical.** No GET→POST conversions.
3. **Auth class identical.** A route currently behind `require_role(["FeedCreator"])`
   stays behind the equivalent v2 capability (`feed.create` per
   [`04-authz-model.md` §3](../../docs/architecture/v2/04-authz-model.md)). The
   `QuizManager` bypass is a known wart — preserve it in v1-parity smoke until
   Phase 2b cuts over to the new RBAC.
4. **Status code on unauth identical.** API/JSON paths → 401. Page paths → 302.
   (See `auth.require_role` early-return.) The smoke test asserts both classes.
5. **Response shape (top-level keys) identical.** `api-feed.json` always has
   `{"feed": [...]}`. `api-course-chapters.json` always has `{"chapters":[...]}`.
   `api-course-framework-explainer.json` always has `{$schema, masthead, parts,
   code, coder, ...}` at minimum.
6. **Filesystem fallback for `/api/course/framework-explainer` preserved.**
   This is the only route with a DB-then-disk source-order; Phase 2a must keep
   it until the ETL is canonical (see `03-data-model.md` §2.6 and
   `01-blueprint.md` §5).
7. **Range-request semantics on `/media/video/{asset_id}`.** Phase 2c
   (decommissioning large objects) must not regress 206 Partial Content; the
   smoke test pins the contract.

## Known gaps the smoke test does *not* yet pin (logged for later)

- `POST /quiz/start` → `POST /quiz/submit` — requires a real session. The smoke
  script can only assert the 401 baseline. A logged-in flow test belongs in an
  end-to-end harness, not the baseline parity net. Phase 2a (quiz_sessions
  table, `03-data-model.md` §2.3) is the right time to add it.
- `POST /api/media/upload` — same: real session + a valid sample file. Defer.
- Google OAuth callback — exercised in prod only; mock with `DEV_MODE=true`
  and assert that `/auth/google` 307s to `/login` in dev (already cited).
